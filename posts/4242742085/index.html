使用 Cloudflare Pages 托管静态网站

发表于2024-07-08|更新于2024-07-08|经验记录
|字数总计:1.8k|阅读时长:5分钟|阅读量:

这个站点是部署在 GitHub Pages 上的，如你所见，在某些情况下访问速度令人不敢恭维。之前某次因为未知原因 Cloudflare 又一次在国内受到“特别关照”，不得已关了一些功能。最近感觉站点访问速度出奇地慢，换了 CDN 也未见明显好转。

一开始在选择部署平台时便对 CF Pages 有耳闻，由于顾虑而未使用。近日尝试，发现其速度确实不错，但晚上得有点难看。

由于某些原因，写作本文时图床工具出现问题，暂不添加图片，故一些操作流程会写得比较详细（甚至有些多余？）。

区别

注：Pages 首页 和
Pages 页面 -> 限制 有更详细的说明

与 GitHub Pages 比较（部分）
比较项GitHub Pages（自定义域名，启用 CF 代理）Cloudflare Pages
特点功能相对简单，只支持静态页面，但管理较方便配置和管理稍显复杂，可以动态部署（类似于 GitHub Action 但更稳定）；可以连接 Git 仓库（自己账户下无论公开与否），能自动检测仓库变动触发更新（可设置关闭）；有更多可配置项；没有 ban 百度爬虫（但某度爬虫确实比较猛）
速度对于小型网站来说速度不错，复杂网站和大流量情况下速度较慢CF 全球 CDN 加速，可以提供更快的网站访问速度，通常情况下比 GitHub Pages 快
免费版限制Pages 仓库须设置为公开；每小时最多 10 次提交每月最多 500 次提交；只能同时部署一个 Pages；生成将在 20 分钟后超时；站点最多可包含 20,000 个文件，单文件最大 25 MB；单个账户最多 100 个项目
访问限制【普通使用一般达不到限制条件】无限请求（与 Worker 不同，Worker 有 100,000 次请求限制），无限带宽（现在有没有调整我暂不知晓）

对 Functions 的请求按 Cloudflare Workers 请求计费。可以在 Workers 文档中找到 Workers 计划和定价。

对 Pages 函数的请求将计入 Workers 付费计划的配额，包括从函数到 KV 或 Durable Object 绑定的请求。

在免费和付费计划中，对静态资产的请求都是免费且无限制的。当请求不调用函数时，它被视为静态请求

免费计划将 Workers 和 Pages Functions 的请求合并。对 Pages 上的静态资产的请求将继续免费，且无数量限制。也就是说，如果在 Pages 上部署的是静态网站，不调用函数，那么它于你而言约等于无限使用。

准备工作与提醒

你需要准备：

  • GitHub 账户
  • Cloudflare 账户
  • 一枚域名且由 Cloudflare 管理（不求稳定性可以不要）
  • 拥有如何正确使用搜索引擎的能力

提醒：

  • Worker 不止能用于部署静态页面（譬如前面那篇部署 BingAI 的）
  • 不知什么原因，.pages.dev访问速度尚可，绑定域名后就不一定了，这可能少数情况
  • 国内 CDN 效果不明显（有时甚至还“倒退”了）
  • 鉴于已有不少人拿 Cloudflare 做黑灰产，所以它的 IP 未经优选稳定性未知，故不要指望这么一做就能让你的网站速度媲美国内云厂商的。
  • 如果条件允许且业务重要，建议使用付费版本；主要业务在国内的建议正规备案、使用国内大厂服务器，那速度绝对起飞

开始部署

该方法更适合部署纯静态网站。虽然它本身支持一些框架，但没有 Hexo（和隔壁 Vercel 一样）。如果你想部署静态页面或使用的框架不被支持，则按照这一条走。

打开 Cloudflare 控制台，进入主页面（此时不需要进入下面的域名管理页面，如果没添加自己域名请忽视这条提示），在左侧菜单下滑进入Workers 和 Pages->概述->创建应用程序，选择Pages选项卡，在通过导入现有 Git 存储库创建这行字下面点击蓝色连接到 Git按钮。

进入从您的帐户部署站点页面。由于我的站点文件托管于 GitHub，因此在该页面选择GitHub选项卡并选择 GitHub 帐户与储存库。如果未显示存储库，请按照网页提示安装 GitHub APP（可选择全局安装或指定仓库安装）并在 GitHub 上为 Cloudflare Pages 应用配置存储库访问权限，等待页面跳转（没跳转的手动返回）。如果安装成功，这里会正常显示你的目标仓库名称。选中后点击右下角蓝色开始设置按钮。

进入设置构建和部署页面。在这里你会看到这些配置项：

配置项含义
项目名称不一定与前面选择的 Git 仓库（例如前面选择的 GitHub 仓库，下同）同名。此处的填写将与默认域名有关，例如填写xxx则部署完后提供的默认域名为xxx.pages.dev。部署完后貌似不能修改这里的xxx，改默认域名可以删除项目后重新部署
生产分支与 Git 仓库分支相同。如果你将文件推送到 Git 仓库的 main 分支（通常这是默认分支）且你希望直接展示的是 main 分支下的文件，那么这里你应该选择 main。其他分支将被部署在预览环境
框架预设如果你想让它原模原样拉取仓库，那么这里请选择 None
构建命令如果你想让它原模原样拉取仓库，那么这里你可以填写exit 0
构建输出目录默认为根目录，对应 Git 仓库的根目录

如果你正巧使用 Hexo 且只推送了源文件而非渲染后的文件，最后三项你可以这么填：

配置项填写内容
框架预设None
构建命令hexo clean&&hexo generate
构建输出目录public

如果你不想要 Pages 默认 Node 版本，则还需点击下方蓝色环境变量（高级）字样，点击蓝框+ 添加变量按钮，填写左右两个文本框，左侧为NODE_VERSION，右侧为三段式版本号，填完后应类似于NODE_VERSION=20.11.1。保存，然后点击右下角蓝色保存并部署按钮。

进入构建和部署页面。等待执行完成后，回到Workers 和 Pages->概述，进入你刚才部署的 Pages，等待一会儿，便可以使用它提供的【项目名称】.pages.dev域名访问了。

自定义

实际上，你可以把 Pages 理解成特殊的 Worker，一些配置项可以互相借鉴，故此处一带而过。

如果你希望后期更改环境变量（例如 Node 版本），请在该项目主页进入设置选项卡，找到环境变量进行设置。

在设置选项卡构建与部署选项中点击分支部署控制->生产分支->配置生产部署按钮，将启用自动生产分支部署取消打钩，可取消自动部署。
