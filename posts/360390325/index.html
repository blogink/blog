请问您今天要来点 FOSS 吗

发表于2026-03-10|更新于2026-03-10|绘览三千
|总字数:7k|阅读时长:21分钟|浏览量:

上一篇拖得太久了——难忘的事不是每天都有的，不过以我的行动力也难说能否一字不差地记录下来（捂脸）。而且，就平常琐事，也不见得每次都有什么大道理可讲（硬要升华的话就和以前写作文那样了）。

因此，我决定加入一些非日常的内容……

自由软件与开源软件

按照 GNU 里面的说法，自由软件的定义如下：

“Free software” means software that respects users’ freedom and community. Roughly, it means that the users have the freedom to run, copy, distribute, study, change and improve the software. Thus, “free software” is a matter of liberty, not price.（“自由软件”尊重用户的自由，并且尊重整个社区。粗略来讲，一个软件如果是自由软件，这意味着用户可以自由地运行，拷贝，分发，学习，修改并改进该软件。因此，“自由软件”是关乎自由的问题，与价格无关。）

因此，它更侧重控制权，无关乎价格（自由软件可以是商业软件，甚至必须可以商用、可以商业开发、可以商业发布），只不过许多开发者都免费提供它们。只要不违背这些核心自由，自由软件许可证也可以额外添加一些要求，限制软件的分发步骤，例如“著佐权”便是常见一种的限制（在 GNU 项目中，我们使用 copyleft 这一规则来保护软件用户的四大自由，并要求软件的修改版必须同样以自由软件的形式发布）。

那么与之相近的“开源软件”，其判断依据是源代码是否向公众开放，是否允许所有人查看、修改、再次分发，但重点在于代码共享、协作、实用性。开源软件只是在软件发行的时候被要求允许用户自由更改、衍生开发，且同样可以是商业软件（例如收费提供服务、定制）。

从这里可以看出来，这两种软件都是开放源码的，都可能收费。二者不同点在于自由软件更在意用户的使用、修改、分发的自由权利，而开源软件更在意代码公开性、协作开发方便性。

至于“免费软件”，那就不在于是否开源了，仅仅只是（暂时）不收费，或是基础服务不收费。典型的例子就是 QQ、微信。

GNU 网站上认为自由软件比开源软件要好，是从理念的角度出发的：

As one person put it, “Open source is a development methodology; free software is a social movement.” For the Open Source movement, nonfree software is a suboptimal solution. For the Free Software movement, nonfree software is a social problem and free software is the solution.（就开源运动而言，软件是否应该开源是一个实际的问题，而不是道德诉求。正如有人指出，“开源是一种开发的方法；自由软件是一场社会运动。”对开源运动来说，非自由软件不是最佳答案。对自由软件运动来说，非自由软件是社会问题，而自由软件是正解。）

它指出自由软件运动和开源运动的基本原则不同，但在一些方面相近，而且自由软件不把开源软件当敌人（按它的说法，敌人是“专属软件”），所以两者能够共同成就特定的项目。

Free Software

在英语中，“free”能指“免费”也能指“自由”，GNU 认为这里应该解释为“给予使用者自由的软件”，认为“免费可得的软件”和“opensource software”都不合适。

GNU 认为，开源软件在某些方面稍显松散，而且它接受的某些许可证给用户带来限制是我们不能接受的；“可查看源代码”这一标准比自由软件弱太多，而且它还可以包括专属软件。

而自由软件有四大特点：

  • 用户可以随意运行软件，用于任何合法场景，不受软件方的用途限制
  • 用户可以自由获取软件的源代码，自主查看软件的底层逻辑，不存在技术黑箱
  • 用户可以自由修改源代码，根据自身需求调整软件功能，适配个性化使用场景
  • 用户被要求分发原版或修改后的软件，既可以免费分享，也可合理收费，传播权完全归属于用户而非单一开发商（不允许被人改成闭源私有软件）

总而言之，在普通闭源软件里，用户只有使用权，没有修改、分发、探究底层逻辑的权利，软件的规则完全由开发商制定；而自由软件把话语权交还给用户，让软件从“开发商管控的工具”变成“用户可自主掌控的工具”。

在二十世纪七十年代的美国，一个人工智能实验室使用一个 ITS 分时操作系统。开发人员用汇编语言为 Digital 的 PDP-10 计算机设计和编写程序，我们的自由软件之父 Richard. M. Stallman 的工作就是改进这个系统。

自由拷贝和源代码开放是整个计算机业的软件开发和传播模式，软件是鼓励自由拷贝的，如果你看到别人使用一种你没见过且有意思的软件，就可以向他索要程序，这样你就可以读它、修改它，或者拆卸部分用于新的程序。早期计算机多是大型机，软件和硬件一起售卖，源代码公开共享是常态。

然而到了八十年代，软件行业从“共享协作”转向“专有垄断”，什么“版权”、“许可证”、“授权”等就够许多用户喝一壶了。好多有才能的 MIT 程序员投入了市场的怀抱，进一步破坏了相互协作、彼此交流的社区。Stallman 感到一个时代结束了，感到很难过。他是 MIT 实验室的程序员，因为无法修改一款调试工具来解决自己的问题，感到被软件束缚、失去主动权。他认为，软件应该是服务人的工具，用户有权查看、修改、分享软件，而不是被软件 “控制”。这份不满，成了自由软件产生的直接动力。

“因此，我开始寻找一条出路，使程序员可以做真正的好事。我问自己，我能写什么软件？我能否让社区重焕生机？”他还在自传中表示“在我的记忆里，Richard Stallman 是我的俗名，人们可以叫我 RMS。”

到了 1983 年，Stallman 在 net.unix-wizards 新闻组上公布公开发起 GNU 计划，宣布它的目标是创建一套完全自由的操作系统，并附带一份《GNU 宣言》，其中一个重要理由是要“重现当年软件界合作互助的团结精神”。GNU 是“GNU’s Not Unix”的递归缩写，Stallman 宣布 GNU 应当发音为 Guh-NOO 以避免与 new 这个单词混淆。

后来他辞去了 MIT 的工作（他担心 MIT 会改变产品的所有权，与他的理念不符），投身 GNU 工程计划，并采用了部分当时已经可自由使用的软件。后来，便是我们所熟知的 GCC 编译器、GNU 通用公共协议证书（GNU General Public License）、Emacs、自由软件基金会（Free Software Foundation Inc）等等——以及 GNU/Linux。

Linux 是他的计划中第一个真正可用的自由操作系统。从此，自由软件从“理想”变成了“可用的工具”，到今天被广泛用于服务器、手机、嵌入式设备等场景。

它打破了“软件必须闭源收费”的垄断，让协作开发成为主流，为后来的开源运动奠定了基础，影响了互联网基础设施、移动系统、编程语言等几乎所有领域。至今，全球仍有无数开发者甚至商业企业参与 GNU 项目，持续维护和开发自由软件，践行“用户自由”的理念。

有些专有软件貌似不是非用不可的

一切的开始要追溯到 2020 年。那时候疫情刚开始，学校所有的课程都转到了线上。对于很多学生来说，那段时间的生活节奏被打乱了；对于我来说，计算机从一个娱乐工具变成了必需品。

以前用平板都主要是看电影、玩游戏，或者写写文档。电脑？什么是电脑？哦那大铁箱子就是电脑啊？（此前我对其认识真就是这样，甚至是看到蓝屏就吓哭了）。好吧，网课期间，正课还用手机平板听实在有点“坐牢”了，而且家中也有条件，我便使用了电脑听课（那个笔记本最初是拿来炒股的，但后来被我“征用”了，谁知它一直陪我到了高考）。

我需要用它上课，做作业，提交资料，还要和老师同学沟通。使用频率的增加，让我开始注意到电脑里运行的各种软件——好像没有想象中那么困难。值得一提的是，我不在乎某个软件是谁开发的，也不会在意它是怎么运行的，只要它能够帮助我实现我的目的，那么它就是“好”。

现在想来，以前算是心大，甚至因为网盘空间充足而把所有能想到的东西都备份上去，然后本地清空省下空间（放到现在是我不能忍受的）。不久后，便因为某些事情让我对“隐私”一类的东西逐渐重视起来，又接触了一些可以自行部署的软件（最早接触的是网盘系统和 Minecraft 服务器）——我对“开源”和所谓“自己可控”的了解便是从这里开始。又因为此时学校上网课，那么电脑接触的也多，也接触了一些过去觉得十分高大上的东西——包括现在这个网站。

也是那个时候忽然发现，“网站”好像也不是什么神秘的东西；如果条件允许，个人搭建从技术上来说没想象中那么困难。网络上有很多教程，教人如何使用静态网站生成器，或者如何搭建 WordPress 博客。我跟着教程一步步操作，购买域名，租用服务器，安装程序。我随即便搜寻了些如“免费主机”的网站（没钱），登录、注册、配置 FTP、上传文件等等。输入地址，当我看见我上传的页面真的被加载出来时，我的内心无比激动（原谅我想不出好词了）。

那么问题也就在这里：为了让网站好看一些，我需要找一些图片，也需要选择合适的字体，和以往一样是看到喜欢的就下载下来用；就这样，直到某日在一个技术论坛上看到了一篇关于“商业授权”的讨论帖。

那篇帖子讲的是有人因为在自己的项目中使用了一款看似免费的程序，结果收到了律师函（没错就是织梦）。原因是那款程序虽然对个人非商业使用免费，但这个界定标准很模糊，如果触发则需要购买授权。

后来了解了，我发现其实大多数人对于“商业使用”的定义都很模糊（还有些人会故意装作普通用户发布商用字体来“钓鱼”），只是程度不同和追究与否的问题。那现在看肯定不觉得奇怪。也许是做些小组作业等内容时还远不及“商业使用”的标准，因此用了也无妨；但在那时，我方才意识到这不是想用就用的。

倒是让我想起不久前往网盘里猛存文件的事情。那些所谓“用户协议”、“使用许可”、“隐私权限”等，倘若细细看来，实在有些“恐怖”——尤其是商业公司的。虽然说“不做亏心事，不怕鬼敲门”，但谁知道我的数据会被怎么使用呢？本来在如今信息发达的时代讨论“隐私权”就显得“矫情”，尤其是到了近几年人工智能突飞猛进的时候更是，但想到自己的资料在不知不觉间被不知什么人以不知何种目的访问，内心还是有些不自在。

这件事对我的触动很大。到底有多少软件是真正免费的？我们日常使用的操作系统、办公软件、甚至一些小小的工具，它们的背后到底有什么样的限制？为了解开这个疑惑，我开始有意识地去查看 （以前被我视作废话的） 软件的使用协议，从中能分辨出一些关键词，比如“专有软件”、“闭源”、“禁止反向工程”、“仅限个人使用”等等。

当我开始关注开源软件后，也便认识了 GitHub，也逐渐理清了“自由软件”和“开源软件”这两个概念。所谓自由软件更强调用户的自由，强调你有权使用、研究、修改和分发软件；开源则能让我获取其源代码。即使我不会写代码，但我可以知道这个软件是怎么运作的，可以向社区里的开发者反馈和提问，而不是只能等待企业慢悠悠地更新些不痛不痒的东西（还顺带塞了些理论上不该有的东西）。

虽然以我目前的水平远远不及那些专业的开发者，但使用这些软件，我所关心的几个问题变得明朗了：我能否掌控我使用的工具，我的数据是否安全，我是否会被厂商绑定。也是从这里开始，我认识到并非所有应用都是企业所为，并非所有应用都是“必须闭源收费”的。而且它们的透明度很高，有无数的开发者“盯着”，一有问题便有人响应并给出建议（甚至直接提交代码）；哪怕原开发者“作妖”，社区也能开辟分支（例如 MinIO、Alist 等）。他们所追求的好像不是盈利（虽然也有而且“自由软件”不排斥收费），但这种平等自由的开发与使用理念，我很欣赏。

如今，随着对软件环境的掌控欲增强，除了非用不可或迁移成本太高的情况，我已经尽可能将软件用自由软件或开源软件替代；有一些服务性质的工具，我可能没有使用，但若有基本可用的自由软件或开源方案，我也会收藏一份。为了一个驱动而翻阅论坛，为了一个字符偏移而研究几日……如此，连动手能力也培养起来了，“提问的智慧”也逐渐学习了——甚至，还能帮助解答别的用户的问题。

当你发现一个工具正好符合你的需求，而且它尊重你的自由时，那种满足感是单纯的消费无法带来的。也许这个观念不是所有人都赞成的，而且要完全取代原来的专有软件也不大可能（国内例如微信、QQ、支付宝等，如今没它们几乎寸步难行）。然而，这意味着，面对工具时，我不再只有唯一的选择了。

Hello Linux

现在的 Linux 生态是自由软件、开源软件、专有软件混在一起的。再来回顾一下：

  • 自由软件一定开源，但开源软件不一定是自由软件。
  • 此三者都可以收费、可以商用，和免费无关。

Linux 内核本身（GPLv2 协议）便是一个典型的自由软件；此外还有一些系统自带工具（Bash 终端、GCC 编译器、Vim）、不符桌面软件（LibreOffice、GIMP）。部分 Linux 发行版（例如 Debian）也算自由软件，但默认完全不包含任何专有驱动、专有软件（譬如在编辑 Debian 的sources.list时需要加个参数来允许使用非自由软件）。

相比于此，开源软件不强制用户修改后必须开源，也不强调“用户自由”的理念。例如 FFmpeg、XFCE 等，必须遵守其开源协议，但不一定遵守自由软件的要求。

还有一类是源码完全不公开、用户只有使用权的软件，即专有软件。例如 NVIDIA 的 Linux 驱动（当时 NVIDIA 的 Linux 驱动很不靠谱）（Facebook 的 pytorch，谷歌的 Android，既是开源项目也都是他们真的在用的；但 NVIDIA 搞的东西多为闭源，虽然现在有所好转）（So NVIDIA, Fuck You!）、微信的 Linux 版（我印象中 tx 也有不少开源的真东西，但有些闭源和开源产品有所谓“霸王条款”）、JetBrains 的部分开发工具。

如上一节所言，Windows 也是个专有软件，而且还要授权（只是基础牢固而且平常也不管盗版罢了），而且还是闭源的（有着无数的 Bug 和无数不知有没有、何时有、如何有的“后门”）。相比而言，Linux 显得“自由”多了，而且能给的权限也很高（拿个经典的例子来说，Linux 用 root 权限可以“自杀”并“善后”，Window 能给自己的 C 盘清空吗）；虽然门槛和生态不如 Windows，但图形化界面方案降低了它的准入门槛，再加上最近几年它的发展也好起来了（我有理由怀疑这是对 Windows 11 大失所望后“叛变”的）、兼容性方案也愈发完善（Wine 11 便是个很好的例子，它甚至还把游戏性能也优化了）。至少在轻度办公方面，它的体验也是好起来了（“4GB RAM on Linux vs. 16GB RAM on Windows”）。

然而，我不是“轻度用户”，现在还是拿 Windows 作为主力机，Linux 作为开发和实验用。然而，在面对一些需要更高“安全性”的场景下，我不至于只有 Windows 这唯一的选项了。

So, Snap, now you get out of here fast!

（我承认我是为了这碟醋包的饺子。）

最近心血来潮，想装个 Linux 虚拟机运行些容器化的项目（多为 AI 相关的项目）。大致看了一下，CentOS 最稳定的版本停止维护了，FreeBSD 不认识……那就用 Debian 吧，也宣称是为服务器优化的。话说 Ubuntu 是基于 Debian 开发的，其他还有些出名的 Linux 发行版又是基于 Ubuntu 改装的，那么顺带再装个 Ubuntu，当作家用系统试试水（话说微软在 Windows 11 作妖无数的同时，Linux 中 Wine 11 发布了，而且对游戏的支持进一步增强）。

然后我忽然想起来：Ubuntu 好像也做了些不受欢迎的事情。装完一看果然，大部分默认程序用的是 Snap 包。

找了找，GitHub 上有这样一个项目：popey/unsnap。它的作用是“Quickly migrate from using snap packages to flatpaks（快速从 snap 包迁移到 flatpak）.”

后来 Ubuntu 的 Snap 包越来越不受到欢迎，Linux Mint 后来甚至默认禁止了 Snap：

Snap 为什么不受欢迎

Following the decision made by Canonical to replace parts of APT with Snap and have the Ubuntu Store install itself without users knowledge or consent, the Snap Store is forbidden to be installed by APT in Linux Mint 20.

无论是哪个 Linux，安装软件基本上有以下几种方式：直接从源码编译、安装编译好的包、使用包管理器（处理对环境依赖要求高的程序时很方便）……但这些方法都有一个问题：不够通用。

这也不难理解，都是基于 Linux 内核，但 Linux 主流发行版又不只有一个，更何况还有“非主流”的；包管理器也不一样，Debian 系和 Red Hat 系的包管理器命令也不同，Arch Linux 也有自己的命令……

所以，Linux 在后来引入了“通用包”，有 AppImage、Flatpak、Snap 等；社区也推出了名为 WeLees Unified Packager（或者叫它 uni-pkg）的工具来统一命令，减轻不同发行版中包管理命令不同带来的问题。不过这里我们重点说说前者。

这种通用包自然而然是通用的，几乎支持所有 Linux 发行版。它的实现方式是把配套依赖全部集成到软件里（和那些 Electron 应用内置 Chromium 类似），虽然占的空间多了，但通用性更好。

其中，Snap 便是 Canonical，也就是 Ubuntu 的母公司开发的，也自然被预装在 Ubuntu 中发布。由于 Ubuntu 在 Linux 中十分知名，用户数量和社区生态也相当可观，因此 Snap 也便推广开了——至少 Canonical 当初是这么想的。

结果？社区表示十分不满。

既然 Ubuntu 是基于 Debian 开发的，那么 Ubuntu 也能用 apt 包管理器，也承诺不会移除它——当然它一直都可以，现在也是。

但 Canonical 是一个商业公司，它怎么会不想方设法利用这个知名产品推广自己的服务呢？商业公司产品有偏好很正常吧？后来 Ubuntu 自带的应用商店便是 Snap 商店，里面下载的应用都是 Snap 包而不是 deb 包。

它的 Snap 包效率还不行，Snap 采用沙盒加自包含打包，每个包都带完整依赖与运行时，冷启动时需挂载解压，比 deb 慢很多，磁盘占用也更高。比如 Chromium Snap 版启动耗时明显长于 deb 版，LibreOffice 等程序更明显。由于沙盒隔离，导致 Snap 应用与系统桌面融合度低、文件对话框不默认关联系统目录、主题不一致、快捷键冲突等等，日常使用有割裂感。

嗯，虽然这 Snap 包的执行效率和占用体积比 deb 难看得多，但大不了我不装 Snap 应用不就行了吗？

在一些教用户去除 Snap 的教程中，可以注意到它有一个默认配置，它会提高 Snap 的优先权。不过，在用户不知情的情况下自作主张，在 Linux 中算越界了。同时，Snap 商店只有 Canonical 管理，无法建设私有商店，依赖其商店与服务，变相形成生态绑定；Snap 包强制更新，且用户无法关闭单个应用的更新。Snap 是开源技术，但分发环节闭源且被中心化管理，它更像是一个“专有软件”，与 Linux 的理念相违背了。

接着前面的“自由软件”，Linux 每个发行版本都始终遵守开源与自由的理念，因此你可以随心所欲的切换任何一个你喜欢的系统，不会被限制。Red Hat 也是个商业公司，但它的 yum 包管理是开源和自由的，任何组织或个人都可以搭建私有的应用仓库。Red Hat 也有商业服务，但它对 Linux 生态的贡献也大于 Canonical，而且在产品上十分克制，遵守自由与开源的理念；相比之下，Canonical 强行推广 Snap 的行为，有点忘本了。
