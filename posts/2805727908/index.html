构建独属于自己的信息流：FreshRSS 上手体验

发表于2026-04-15|更新于2026-04-15|体验推荐
|总字数:3.6k|阅读时长:10分钟|浏览量:

如今各种平台的推荐功能和信息流已经很完善了，但我认为这仍然是一种“被动”为主的接收信息方式。于是我整理了许多网站作为我的信息流来源。虽然它们不一定涵盖方方面面，但至少都是我自己选出来的，也算是有一定的主动性了。

煮啵煮啵，你的平台推送确实很强，但逐个访问还是太吃操作了。有没有一种技术可以将这些网站的更新聚合起来、统一获取以供阅读，并且减少平台推荐算法的干扰？

RSS 服务

有的兄弟，有的。让我们有请一个诞生于 1999 年的技术——RSS（Really Simple Syndication/RDF Site Summary）。

简单来说，这是站点用来和其他站点之间共享内容的一种简易方式。用户自己说了算，只要这个网站提供的订阅链接还可以访问，那么用户想什么时候以怎样的方式访问就由其自己说了算。

你也可以看到，RSS 有多种缩写含义，这是因为 RSS 技术在发展过程中被不同组织作了不同的解释，但它们都遵循 XML 1.0 规范。虽然如今 RSS 已经式微了（它的一个重要作用就是方便了用户去访问网站更新，但现在社交媒体比这玩意更方便），Google 也把自己维护的 Google Reader 服务关闭了。

不过，它的优点放在现在仍有价值，至今仍是开放网络的重要部分。由于它们的内容由网站直接提供，不依赖中心平台分发，使得用户可以自由搭配属于自己的信息源，是早期互联网精神的一种体现。

但随后我又遇到了一个问题。若来源多而分散，我若逐个访问实在耗时（还可能遇上当时网站没更新的情况，白跑一趟）。同时，如果有些文章值得反复阅读或将来可参考，收藏同步也是需要考虑的（Edge 浏览器在桌面和移动端的账号同步似乎有点小问题）。我需要一个能统一管理这些信息聚合的平台工具。

我先前注册了一个免费额度可观的 WebDAV 储存账户，想着这空间十分充足，于是我尝试使用 WebDAV 来同步订阅源和收藏栏。可惜那个 WebDAV 服务是在国外的，国内直连有时会失败，用了一段时间就作罢。还有一个原因是阅读器软件——确实有跨平台的，但既是跨平台又内置 WebDAV 支持的软件我几乎没找到，有的大多也不符合我使用习惯。

既然在软件客户端层面不好解决，那么有没有别的方法？大部分 RSS 阅读器都支持第三方服务同步，常见的有 Fever API 和 Google Reader API。想到以前还注册过一个主机，于是我就想：能不能自己托管一个这样的聚合器，利用它统一管理信息源并提供 API 实现多端同步。

综合考虑易用性和上手难度，我最终选择了 FreshRSS。这个支持自托管的程序已经有超过十年的历史了（另外，有些 RSS 客户端预设的 API 还将其特别注明），想来也是比较完善的。翻看了一下介绍，对部署环境配置要求很低。

（如果要订阅没有提供 RSS 的信息，例如 Bilibili、微信公众号、X 等平台，我们也可以用 RSSHub 来转换，不过这个不在本文的讨论范围内。）

FreshRSS 部署流程

为了尽可能减少部署复杂度和灵活度，很多基于 Python、Node.js 等的项目我只能暂时放弃（大部分主机推荐安装的都是 PHP 环境，许多“免费主机”更是只有 PHP 程序才能方便运行）。

我看许多教程都给出了它的 Docker 部署方式，但实际上只要服务器能跑 PHP 它就能运行，简单使用没必要用 Docker。官方推荐的配置如下：

项目要求
Web 服务器推荐使用 Apache2.4+，也可以用 Nginx 或其他
PHP 环境PHP 8.1+
PHP 必要扩展cURL、DOM、JSON、XML、session、ctype
PHP 推荐扩展PDO_SQLite、GMP、IDN、mbstring、iconv、ZIP、zlib
PHP 数据库扩展PDO_PGSQL 或 PDO_SQLite 或 PDO_MySQL
数据库要求PostgreSQL 10+ 或 SQLite 或 MariaDB 10.6+ 或 MySQL 8.0+

如此一来，只需将 FreshRSS 的 latest 分支下的文件（稳定版） 上传到主机并配置域名解析，就完成大部分前置工作了。

程序提供了 Apache 的管理配置文件，上传主机后只需将.htaccess.dist去掉后缀（即文件名改为.htaccess）。如果用的不是 Apache 作为反代程序也没有关系，最终确保仅向 Web 暴露./p/目录（运行目录）且添加./data/目录的写入权限（但不建议暴露./data/目录）即可。

完成后访问域名，若配置正确则可进入安装页面，根据页面上的指示完成安装即可（可以设置中文），此处不过多展开。

安装完成后登录，主界面大致如下。FreshRSS 会默认订阅官方的更新消息。

添加订阅源与分类

在左上角区域点击订阅管理按钮，进入管理页面，进入订阅管理-添加订阅源或分类区域，在添加订阅源中填写 RSS 链接（通常包含 feed、rss、atom 等字样，需要根据实际情况选择）。除非你想添加的订阅源不是 RSS 或 Atom，或是在意 RSS 订阅输出被站长设置了重定向（例如 RSS 只能输出简介，正文需要跳转回网站访问），否则不用管订阅源类型和高级两个选项。

我们以 阮一峰的网络日志 为例，在页面中找到 RSS 链接（这个因人而异，图标通常是一个像 WI-FI 的符号，进入后多为一串 XML 代码，不排除有人会给它进行美化），将其复制到添加订阅源的源地址输入框，然后点击添加。你也可以在刚才的添加分类和分类中填写或选择分类名称，以后可以将相近类型的信息源分类整理。

完成后跳出横幅订阅源 阮一峰的网络日志 已添加，并自动进入该订阅的详细设置页面。在这里可以进一步调整图标、标题、描述、分类、抓取频率等。

回到首页。一般来说，添加完后它会自动更新订阅源；如果没有的话就点击首页的更新订阅源按钮，等待一会后查看订阅效果。

注：默认情况下，如果有更新则首页只会显示未读文章，全部已读后先前已经获取过的文章缓存（包括分类）才会重新显示在左侧菜单中（可能是出于方便区分已读未读的原因）。

前面我们提到，它有订阅源分类的功能。进入订阅管理面板，选择订阅管理一项，可以管理目前所有的订阅源的分类。你可以通过拖动来改变某一项的分类，也可以点击订阅源和分类标题前的齿轮调整其说明信息（就是之前刚添加完订阅后切换页面的那些配置项）。

API 设置与客户端

在接入客户端之前，需要先启用 API。点击右上角的齿轮按钮，选择认证并输入密码验证，将允许 API 访问一项勾选后提交。完成后在左侧菜单中进入账户，在API 管理中设置 API 密码并提交。随后访问一下旁边提供的地址检查 API 状态。

此处可以看到它提供了几种 API。如果要接入客户端（例如 Fluent Reader、Read You、RSS Guard 等），我们只需记录下 Google Reader compatible API 和 Fever compatible API 两个地址。

下面我们以 Fluent Reader 为例配置接入。点击顶部导航的齿轮按钮进入设置，选择服务选项卡，添加一项服务。我们选择 Fever API，在里面填入刚才的参数：

  • 端点：刚才的 Fever compatible API 地址（注意别漏字符了）
  • 用户名：当前使用的 FreshRSS 账户用户名
  • 密码：刚才在 API 管理中设置的密码，不是账户密码。

填写完成后点击确认并等待更新。如果没有自动分组，需要在分组与排序选项卡中导入分组。

其他客户端按照类似的方法配置（有些客户端中 FreshRSS 有单独的选项），有些客户端可能支持不完善或是多出功能，具体看阅读器本身的特点。我个人感觉 Google Reader API 在同步时效果会好一点。

插件

之前提到可以手动点击主界面的刷新按钮来获取文章更新，但这显得有些麻烦。也可以去定时访问https://www.example.com/p/i/?c=feed&a=actualize（请自行替换实例地址）链接进行更新。实际上 FreshRSS 有扩展可以实现这个功能。

点击右上角齿轮按钮，在展开菜单中找到扩展，可以看到当前 FreshRSS 支持的扩展，需要手动跳转到扩展的页面（大多是 GitHub 仓库）手动下载，按照扩展的文档将下载好的项目按要求上传到 FreshRSS 根目录下./extensions目录中。

我们以 Auto Refresh 扩展为例。点击后将该仓库 Clone 到本地，并将其中的xExtension-AutoRefresh文件夹上传到./extensions目录中。随后返回扩展页面并刷新，确认用户扩展中有自己的目标扩展（若是可用的社区扩展，则这部分表格中会有已安装字样），将该扩展的复选框（此处即开关）选中，即可启用。

部分扩展可以自行配置，点击开关旁边的齿轮按钮即可进入管理页面。

使用体验

FreshRSS 解决了我用 RSS 时一个很重要的问题，那就是聚合管理。（也许是我没设置好，在有些平台上它只会同步未读的，有些平台会把所有获取到的文章都同步下来，无论是否已读）

我的需求很简单，关注一些常看的资讯网站，也有几个个人或团队网站，不用逐个打开访问就能看到所有更新。可之前没找到合适的工具，只能在不同平台之间来回切换——这样一来，同步的问题就显现了。诚然直接访问大平台更加方便，而且不会局限于单一领域，但我认为大平台总是要“殊途同归”的，最终还得让个人或团队网站来补齐一些非流量驱动且有价值的内容。

现在 FreshRSS 最让我满意的，就是它完美解决了同步的痛点，而且部署难度不大。不管是什么设备、什么软件，只要能支持登录同一个账号，那么我所有的阅读状态都能同步（发展到现在，它本身的 Web 界面也相当完善，换言之我甚至可以只要个不太旧的浏览器就能看更新）。

此外，它可以协助我统一管理订阅源，把所有的订阅源都集中在一个界面里，只要有 RSS 链接就都能添加进去；没链接的，若用其他开发者的转换工具（例如 RSSHub）或是 FreshRSS 自己的抓取工具，也能添加上（有些平台的反爬措施愈发严格，这只能暂且作罢）。

试着使用了两三天，我将原来分散在各处的订阅配置都导出并添加到这里。久而久之，我整理了属于自己的信息源，没有平台算法干预（除非它在 RSS 输出中截断正文，这也是存在的），所有的内容都是按照我筛选的订阅源的更新时间排序。说到底这只是一个工具，内容需要靠自己或参考他人推荐来添加；除了“统一同步”这个程序优点之外，其他的优点多为 RSS 的优点。

关于隐私问题，虽然在现在这个科技发展阶段下讨论隐私有点奢侈甚至算妄想了，但 RSS 可以很大程度上减少“猜你喜欢”这种的影响。像什么阅读习惯、浏览记录甚至是个人信息，在 RSS 中能起到的作用几乎没有（除非有那种注册登录填写 Token 才能生成订阅地址的平台），几乎不用担心隐私泄露的问题。

只是，在现在的互联网上，选择给“开放”设门槛的越来越多了。无论是要跳转下载 APP 才能正常访问的残缺 Web 页面，还是压根不给搜索引擎多少收录机会的集群（例如微信公众号），RSS 能抓取到的东西也不一定那么完善。有些信息源即使订阅了，在阅读器中也只能看到标题和摘要；想看全文还需要点击链接跳转回原网站。

平台推送往往会优先选择那些点击率高、停留时间长的内容，而这大多是标题党、情绪煽动性的内容，质量参差不齐，而且我不知道什么时候会碰到它们。现在在这个程序里的订阅源，都是我自己筛选的，不用现场在一堆垃圾信息里筛选有用的内容。尽管也可能存在茧房和上手复杂度的问题，不过掌握在自己手中的主动权更多了。

我也许会一直用下去，它已经融入了我的日常，成为了我获取信息的重要工具。我也希望 FreshRSS 能一直保持这种简单实用的风格，不要添加太多不必要的功能。
