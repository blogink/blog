Between graphics and text

发表于2026-03-21|更新于2026-03-21|闲聊随写
|总字数:6.2k|阅读时长:18分钟|浏览量:

在经验记录时，图片是个很好的形式。图片当然是非常直观的，倘若我需要讲述如何在网页上点击目标链接，我可以截图并画箭头，这比只靠语言“先点击……再点击……找到……”要直观得多。于是我打开画图软件，拉了几个方框和箭头，调整颜色和字号，最后导出 PNG，上传到图床，然后复制链接，插入文章。仔细一看，发现箭头序号标错了——算了，重来一遍。

不过，总的来看依旧优于文字表述，找相对位置也直观。后来还找了个在线白板工具，可以绘制流程图，也可以插入图片，更加方便了。还是照例，编辑图片、画线、插入文字，然后导出、上传图床、复制链接并插入……文字太干了看着累，放几张图片便显得丰富了。

可惜，现在再看，除了记录某些软件操作步骤（更多是目标结果的画面）不得不留截图外，大部分时候图片反而干扰了阅读。而且有些图成了摆设，尤其是某些更新后操作界面发生较大改动的（例如 Cloudflare 的控制台，许多入口都改了位置）。那怎么办呢？如果要保持准确性，这些图片也便废弃了，甚至成为了干扰项，需要重新做一遍——真的麻烦。

那我一开始为什么要用图片啊！

阅读干扰的问题

图片若放的位置不合适，还容易打断阅读节奏，尤其是长文。读长文的时候，读者的注意力是集中在文字上的，如果频繁插入图片，读者需要停下来看图片，再回头看文字（要我说图片必然比文字更吸引注意力）。

既然是注重用文字表达思想或观念的文章，那么重点是在文字，而图片在文章中扮演着“辅助者”的角色。它既能吸引读者的注意力，让文章更生动、更直观，也可能会干扰读者的阅读节奏，影响阅读体验。在这个信息碎片化的时代，包括我在内许多人的注意力很难长时间集中。一篇全是文字的长文，很容易让读者感到枯燥，若适当插入一些图片能缓解这个枯燥感。

一篇讲解旅游攻略的文章，插入一些当地的风景照片，能让读者更直观地了解当地的景色（我深知我目前的水平还没到写出“飞流直下三千尺，疑是银河落九天”那样的地步）；一篇讲解美食制作的文章，插入一些美食的照片，能帮助读者更清晰地了解美食的制作过程。这种情况下，图片起到了积极的作用，能提升文章的阅读体验。

还有，图片能帮助读者更好地理解文章内容。对于一些复杂的、抽象的内容，文字描述可能很难让读者理解，而一张图片能把抽象的内容变得具体直观，让读者一眼就能看懂。“平静的海面和沙滩形成了对比，三块钟表以柔软布料的样子悬挂在树枝和台面上，旁边还有一只爬满蚂蚁的钟表……”这段话给人的印象远不及把达利的《记忆的永恒》拿来看看来得深。

如果一篇长文每隔几百字就插入一张图片，对读者而言，需要重新找回之前的阅读节奏，有时候甚至会忘记刚才的思路，需要重新翻看。这样一来阅读效率就会大大降低，也会影响对文章内容的理解。还有一些图片，是“装饰性”大于内容作用的，更容易干扰读者的阅读。放在开头结尾还好，若插入正文则不恰当。一篇讲解计算机硬件的文章，若在其中随意地插入一张风景照片，读者会感到困惑，自然也就分散注意力了。

另外，图片的大小、排版、花哨程度也会对阅读造成干扰。我有意识地保持页面中不出现比文字更吸引注意力的非必要元素，并将装饰和个性化设计从图片转移到文字上（例如波浪线、模糊字体、加粗、变色等）。这样一来，即使换了图片，我的“小巧思”还能保留下来。

总之，这是“适度”和“相关”的问题。适度插入与文章内容高度相关的图片，能提升文章的阅读体验；过度插入图片，或者插入与内容无关的图片，就会干扰读者的阅读，影响文章的表达效果。

那么图片多了后，怎么管理呢？

图片管理的负担

开头有些极端了：倒不是从此拒绝在写作中使用图片，只是，我需要考虑某些情况下我是否必须要使用图片。

二进制图像文件给我的管理造成了比较大的影响。在一些情况下，它们占用存储空间远大于同等视觉信息的文本描述，版本控制系统的 diff 对它们无效，迁移或备份时需要特殊处理……当文档数量增长，这些成本累积起来不可忽视。文档里引用的外部图片可能因源站关闭、路径变更、权限调整而无法显示（当然也可以把图片转换成 base64 形式内嵌到文档中，但这仅仅解决了图文分离的问题）。

我经历过几次整理旧文档的麻烦。早期写的技术笔记里插了大量截图，其命名也是随意的，现在想要整理可谓十分困难。如果是用文字描述配合代码示例，即使环境变迁，至少信息骨架还在（要是图片挂了，原本需要靠图片填充的“干涩”的文字也许会更加拙劣）。

这促使我重新评估"必须配图"的决策。很多图只是"当时懒得写清楚"，用现在的眼光看，完全可以用更耐久的形式替代，也不必用 JPG、PNG 等格式的图片。因此，后来我逐渐减少了图片的使用，转而强迫自己要用文字的形式来讲述工作流程乃至日常见闻（那古人写作能“身临其境”，我怎么就不可能呢？……好吧短时间内确实不大可能……）。

在 markdown 中有 mermaid 渲染器，它可以用纯文本的指令，就能绘制出流程图、时序图、思维导图等各种图形。比如我想画一个简单的工作流程，只要用几行文本指令，就能定义出流程的节点、连接关系，渲染之后就能生成清晰的流程图。而且这种纯文本的绘图方式，有很多优点。

如果用图片文件，修改的时候需要打开对应的软件，找到要修改的部分，然后进行调整，操作比较繁琐。而用纯文本来表达，只要修改对应的文本指令，保存之后，渲染出来的图形就会自动更新。比如我想调整流程图中节点的顺序，只要修改文本指令中节点的排列顺序，就能快速完成修改，不用重新绘制整个图形。虽然不如直接用鼠标拖拽画图直观，但它其实更灵活，只是对使用者的抽象思维要求更高；而且纯文本的代码可以很方便地保存和修改，不用担心格式错乱。

其次是占用空间小；若是在网络传输中，则减少加载所需时间。纯文本的指令非常简洁，一个包含几十行指令的 mermaid 代码，占用的存储空间只有几 KB，而一张同样内容的图片文件可就大了（过度压缩破坏了清晰度，有时还不如不放这张图）。尤其是在写长文的时候，如果插入很多图片，会让文章的文件体积变得很大，甚至会破坏阅读的连贯性（你能想象网络不佳时零零散散的文字之间有着大片空白的画面吗）。

还有就是跨设备兼容性好。纯文本的指令可以在任何设备上打开，哪怕没渲染出来也能大概知道它想表达什么。而图片文件可能会因为格式问题在某些设备上无法正常显示（早期 WebP 在 iOS 系统上是无法正常显示的，需要特地适配）。纯文本也不见得简陋，要我说更精确的论文排版用 LaTeX 比 Word 好，虽然麻烦了些。

即便不使用图形，纯文本的描述也能起到很好的辅助作用。很多人觉得，复杂的结构或场景，必须用图片才能描述清楚。其实不然，只要用简洁、准确的文字，同样能把复杂的内容描述清楚，而且有时候，纯文本的描述比图片更有优势。在莱辛的《拉奥孔》中有如下内容：

画只宜于描写静物，诗只宜于叙述动作。

绘画所用的符号是在空间中存在的，自然的；而诗所用的符号却是在时间中存在的，人为的。一个在空间中，一个在时间中。因此，前者宜于描绘物体，后者宜于表现运动。前者通过物体去暗示运动，后者通过运动来暗示物体。

描述狗追人这个事情，需要用许多张图片，但文字只要几个字就行；同样，描述一部画作时，图片可以替代无数的文字。描述一个软件的操作流程，用纯文本分步骤描述，读者可以按照步骤一步步操作（如果他找得到的话）；图片只能展示某个瞬间的操作界面，必要时还需各种箭头、序号，甚至把几张图片拼贴在一起。

当然，纯文本的绘图和描述也有局限性。比如一些需要展示细节、色彩丰富的图形，比如照片、复杂的设计图，纯文本就无法准确呈现。那么干脆做得简洁一点，不要那么花里胡哨的，不就行了吗？但那样的话似乎也没必要特意制作图片了。

如此一来，我只需要很少的占用就能保存最核心的内容了；要用的时候，选择对应的工具打开渲染就行了。只在必要时渲染成需要的形式，可以大大降低资源需求，提高了复用性和灵活性。

“那么，减少图片的使用了，写教程该怎么办呢？这软件操作都写不了了……”

话说，不是所有软件都只能用图案来操作吧……

图形化与命令行

正如开头所言，图形界面非常直观，不用费脑子去记复杂的指令，只要能看懂图标，大致就知道该如何上手了。联想曾经开发了“幸福之家”软件，标语好像是这个：

欢迎来到幸福之家！在这里，您不仅可以娱乐休息、学习办公，还可以通过因特网了解外面的世界呢！

在 90 年代末，电脑在中国家庭刚刚普及，还算是个新鲜事物。这个软件最大的特点就是直观，不管是老人还是小孩，看图标大概能猜到功能。放在当时，这种情景式的互动操作比 Windows95 的图标要明确得多了。

信息呈现也更直观，比如我们看一份数据，纯文本的数字罗列会让人头疼，但画成图表就则一眼就能看出数据的变化趋势。再比如看一篇文章，图文结合的方式能让枯燥的文字变得生动一点，尤其是讲解一些复杂的结构，比如机械零件、房屋空间布局，一张图片比几百字的描述更管用。

图形界面还能降低操作的出错率。以手机为例，Android 系统是基于 Linux 开发的，在大部分人观念中 Linux 是只有一个漆黑的命令行界面的。其实 Android 系统的手机也是可以用命令行来操作的，但不如绿色的电话图标能打电话、信封图标是收消息、相机图标是拍照这样直观。将现实中的物品模拟或抽象化后搬到这里，能给人一种熟悉感，也就省却了需要多费口舌的精力。

但图形界面的缺点也同样明显。首先，GUI（Graphical User Interface，图形用户界面）被渲染出来所需的资源更多。对于图形化操作系统有这样一个调侃，说其只要很少的资源就能完成任务，剩下的资源用来告诉用户它正在做什么、它能做什么。现在许多服务器用的是 Linux 系统，它对资源的要求很低，甚至显卡（包括核显）都不需要就能跑；代价就是这种情况下它没有图形化的界面，需要用命令行来交互。也许一个系统多要一点图形界面的开销没什么，但对于有成千上万项目运行的服务器、无数系统共同运行的服务器集群而言呢？这开销恐怕不能忽视了。

其次，图形界面不够灵活。图形界面的操作都是固定的，它给你提供了什么按钮你就只能做什么操作。如果想实现一些特殊的需求，比如批量处理文件、自定义操作流程，图形界面就很难做到，或者操作起来非常繁琐，要点击很多次鼠标。有些二次开发的程序（例如部分基于 FFmpeg 开发的处理工具）为了保留这种灵活性，选择了只做了基础的图形界面，保留了原来的执行命令的形式。

图形界面的一致性很难保证，单说网页部分，Safari 和 Chromium 的解析逻辑就有些许差异，更何况是其他。图形界面的设计风格、操作逻辑都不一样（你也不能强行要求所有开发者都必须遵守规范）。但是 CLI（Command-Line Interface，命令行界面）就不同了，只需使用键盘就能完成（连鼠标都可以不用）；窗口中都是文字，图片也不必展示出来，描述起来也十分方便。

以 Git for Windows 为例，它提供了图形化的界面（虽然看起来很简陋）。我们以一个十分著名的项目 jQuery 为例，现在我们需要将其 Clone 到本地（例如D:\Projects中jquery目录）。图形化界面如何操作？打开Git GUI，找到Clone Existing Repository，在Source Location如何如何，在Target Directory中如何如何……那么如果我们用命令行呢？

虽然观感比图形化界面差，但若熟悉则更加方便（况且现在 CLI 也不一定是十分单调的，例如 Python 的rich包；能在终端做些原来只能在 GUI 中做的事也不是不行，“无所不在”的贪吃蛇想来不必多言）。

抽象化避免直面底层逻辑

（容我开个玩笑）现在大多数人做开发时都不会还是手工给纸带打孔了吧？各种编程语言其实也是将计算机语言抽象（或者说“翻译”）后呈现出来的。那既然是抽象出来的，肯定也损失了些东西。在计算机编程中，高级语言与低级语言的主要区别在于它们与硬件的抽象层级不同，高级语言使用接近自然语言的语法，与计算机原本的语言差异较大（可想而知编译器的负担有多重）；低级语言效率很高、可以更精准地控制硬件，但编写难度大、可移植性差。

放在图形化界面中，把复杂的操作简化成简单直观的图形和按钮。比如我们点击手机上的“微信”图标，打开微信，输入消息，点击发送。倘若真让用户去学习怎么操作内存、如何做垃圾回收等，微信估计早完了——而图形界面不然。它把这些复杂的底层操作进行了预处理，使得用户不用关心背后发生了什么，只要点击图标就能完成操作。

如此一来它降低了使用门槛，让普通人也能使用计算机；虽说它让用户远离了计算机的底层执行逻辑，但对于普通人来说，只要能正常使用电脑就好了。可一旦想要深入，了解底层执行逻辑就很重要了。比如程序员写代码，需要知道代码是怎么被计算机执行的，知道不同的指令会对应什么样的底层操作。

命令千变万化，几乎不可能为每个命令都去做图形化的适配。如果只停留在图形界面的层面，不知道底层逻辑，就很难写出复杂的程序，也很难解决程序中出现的问题（你可能会说“Scratch 不是图形化编程的吗”，但就我个人看来，不说别的，起码许多大型游戏仍然是使用 C/C++ 等语言来编写的）。在这个方面，CLI 就更接近计算机的执行逻辑了。比如前面的 Clone 命令，每一步都对应着程序的一个操作；相较于图形化界面，没有经过太多的抽象。

还有一点，高度抽象会牺牲一定的灵活性和效率。图形界面为了照顾大多数人的使用习惯，会把操作标准化、简单化，这就导致它无法满足一些特殊的、复杂的需求。而纯文本因为更接近底层逻辑，灵活性更高，可以根据自己的需求，自定义操作指令，实现一些图形界面无法实现的功能。

说到这里我需要补充一下，如今有人评价越来越多的年轻人不懂电脑了（此处的“不懂”是指像不知道打开“我的电脑”指按下电源开关、把快捷方式放到回收站即为“卸载”、认为“手机和平板将来一定会取代电脑”之类的 ）。以“文件系统”为例，现在大部分移动设备弱化了“文件系统”这个概念转而以“沙盒”等方式运行（iOS 早期甚至根本没有文件管理器，Android 虽有像样的文件管理但其实也隐藏了很多东西）；然而 Linux、Windows、MacOS 等常见于桌面设备的系统却不然。也许是因为这个概念对他们而言几乎等于没有，所以第一次从消费品转向生产力工具时会手足无措。相比于以往，现在只用手机就能完成生活中约 90% 的事，其地位在这些人眼中也便没那么高了。

当然，这并不是说高度抽象不好。手机抽象文件系统是其正确的产品决策，这种消费品便是要符合“拿来就用”的逻辑；生产力工具保留了它，是因为生产力工具用户需要更加自由地管理文件。普通人日常使用，图形界面的高度抽象就足够了，不用去纠结底层逻辑；专业的人因为工作需要，了解底层逻辑，使用纯文本操作，能提高工作效率，解决更复杂的问题——所以最终还是“效率”的问题。

如此一来，命令行实在是太完美了……吗？

用命令行一定比用图形界面高级吗

经常能看到有人争论，图形界面和命令行界面到底哪个更好，甚至有人会因为别人用命令行就觉得对方“专业”，用图形界面就觉得对方“小白”；还有人反过来，觉得用命令行是“故作高深”，不如图形界面直观方便。恕我直言，如果在用 VSCode 效率更高更方便的事情上非要选择用 Vim，我会认为这是位“嘉豪”（笑）。

图形界面适合大多数普通人，适合日常的简单操作。比如我们平时聊微信、看视频、逛网页、编辑简单的文档，用图形界面就很方便，不用学习复杂的指令，上手快，操作简单，能高效完成日常需求。对普通人来说，电脑和手机只是工具，能快速解决问题就好，没必要去学习命令行这种相对复杂的操作。

命令行界面则更适合更灵活复杂的使用环境，比如往 SQL 里插入数据，使用命令即可完成；非要用图形化界面去操作，虽然方便，但那样和纯手工操作 Excel 没什么区别了。对于一些固定的、重复的操作，命令行确实更高效（甚至有些实用工具没给官方的 GUI）。

还有人觉得，用命令行的人更“专业”，这种想法很片面。这是个工具，工具是用于实现目的的。所以不在于用什么界面，而在于能不能高效解决问题。一个程序员，用图形界面的 IDE 写代码不见得比用 Vim 写得差；反过来，一个服务器管理员，用命令行解决系统故障，不见得比拿着鼠标操作装了图形化界面 Linux 系统的人要高级。更何况现在很多图形界面的软件的功能非常强大，能满足专业人士的需求。Photoshop、Krita 就是图形界面的软件，使用它能做出非常精美的设计作品；Unity、Blender 也是图形界面，可以做出高质量的模型和动画。这些软件的操作虽然复杂，但效率也都很高。

命令行出现得比图形界面早，它是计算机早期的主要操作方式，而图形界面的出现是为了降低操作门槛、让更多人能使用计算机。两者只是适应不同需求的产物，没有先进和落后之分。比如在服务器领域，命令行仍然是主流，是因为这些程序的灵活性和资源开销使然；而在个人电脑和手机领域，图形界面是主流，因为它更易上手。因此这两者也不存在所谓先进落后之说。

总结

如果要把所有事情都抽象化，不见得是方便的。如果把前面的内容再扩展一下，“问 AI”比“使用搜索引擎”更加抽象和简洁（连“搜索”这件事都可以被精简了）。只使用高度抽象的产品，尤其是在一开始就直接接触它们，会失去了解它们原理的机会，忽略最基础的知识。譬如，同样的 AI 模型，有人能写出完善的程序、精妙的文章，有人连提问都不会，不正是如此？

因此，当其使用门槛下来之后，了解基础知识的人可以利用它自我提高；一知半解又不愿学习的人只能停留在最基本的阶段。不要怕麻烦，这样反而能帮助自己建立清晰认知（大部分学校如果有教“信息技术”的话，会讲到 Python 编程，而在 Python 中有“类与对象”这个概念，只不过大部分时候不会太深入了解。可如果熟练掌握了它，可以推广到其他语言中，即便以后不再经常常接触 Python。）

从长期的角度来看，透过抽象层去接触内核，对培养思维来说（不仅仅是计算机）是十分重要的。
