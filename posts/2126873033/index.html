Hello, Codeberg

发表于2026-04-03|更新于2026-04-03|经验记录
|总字数:4.4k|阅读时长:14分钟|浏览量:

我看这 GitHub 自从被微软收购后也是逐渐开始完蛋了。虽说“百足之虫，死而不僵”，亦或是“瘦死的骆驼比马大”，它的影响力仍然是不可忽视的（上网搜索 Git 托管平台清一色关联到 GitHub，其他的少之又少），但微软的操作以及后来胡乱加入 AI 的行为，成功引起了一些人的不满，例如 Zig。

即使在可预见的未来里我仍需要继续使用 GitHub，但也许我可以找到新的选择。

接下来，让我们有请：Codeberg。

（注：本文不是专门的“经验记录”，如果你是来寻找教程的话可以直接跳过前面长段“铺垫”内容）

Gogs、Gitea、Forgejo

GitHub 和 GitLab 都是商业公司的产品，而 Git 是使用 GPLv2 开源协议的自由软件。Git 项目只是一个命令行工具，但得益于托管服务如 GitHub 等而在 2010 年代初爆火。

那么，有没有办法实现自托管呢？有的兄弟，有的。让我们从 Gogs 开始讲起。

Gogs 是这样介绍自己的：

The painless way to host your own Git service

Gogs（/gɑgz/）项目旨在构建一个简单、稳定且可扩展的自托管 Git 服务，能够以最无痛苦的方式设置。使用 Go 语言开发，这使得其可以通过一个独立的二进制分发程序在 Go 支持的所有平台上完成，包括 Linux、macOS、Windows 和基于 ARM 的系统。

其项目最初由中国开发者 Unknown（又被称“无闻”）与 Lunny 发起，随后又有许多开发者加入团队，共同推动项目发展。Gogs 的首次公开亮相可追溯至 2014 年，其诞生的核心初衷是解决当时主流 Git 托管服务（如 GitLab）资源占用过高、部署复杂的痛点，打造一款“零依赖、易部署、轻量级”的 Git 服务，让个人开发者和中小型团队能够轻松搭建属于自己的私有代码仓库。从它的介绍语中也可以看出，它确实在向这个方向努力。

Gogs 充分发挥 Go 语言编译后单二进制文件、跨平台、低资源占用的优势，将所有核心功能高度集成，让哪怕不懂专业运维的人也能轻松搭建自己的私有代码仓库。无需复杂的依赖配置，即可在 Linux、macOS、Windows 等多种操作系统甚至树莓派这种 ARM 架构上运行。

作为一款开源的自由软件项目，Gogs 采用 MIT 开源协议，任何人都能自由使用与分发。直到现在，Gogs 虽然更新节奏不算快，但一直保持着稳定维护。对于只想安安稳稳用个 Git 仓库，不追求复杂功能的人来说，Gogs 真的足够省心。

但有一个问题，那就是它更多受到“无闻”一人的影响（从更新频率来看，“无闻”应该是一个很佛系的谨慎的人）。它更接近个人主导的项目，对社区的 issue 和 PR 的响应没那么快，而且“无闻”似乎也谨慎参与社区活动，这导致它逐渐显现出“独自掌控”的样子。

嗯……这个做法在“自由软件”中显得非常“危险”，于是 Gogs 的重要贡献者 Lunny 又带领了一批开发者对 Gogs 进行分支，于是 Gitea 便诞生了。它是这么说的：

Private, Fast, Reliable DevOps Platform

和 Gogs 不一样，Gitea 走的是“社区驱动”的路线，大家一起出主意、写代码，目的就是做一款“更易用、更强大、扩展性更好”的轻量级 Git 服务，而且同样坚守 FOSS 的理念，采用 MIT 开源协议，免费开放给所有人使用。

Gitea 虽然是从 Gogs 分叉来的，但后来几乎重写了所有代码，在保留轻量级、易部署优势的基础上，对代码进行了大规模重构和优化，几乎替换了所有原始代码，同时不断丰富功能模块，弥补了 Gogs 的诸多不足。2016 年 12 月，Gitea 发布了 1.0 正式版本，标志着其进入稳定发展阶段。此后，Gitea 凭借活跃的社区支持，保持着快速的迭代节奏，不断推出新功能、优化用户体验，逐步成长为一款兼具轻量级特性和丰富功能的 DevOps 平台。

但是在 2022 年 10 月，Gitea 的核心维护者 Lunny Xiao、techknowlogick 等成立了 Gitea Limited 公司，计划推出基于 Gitea 的商业托管服务，并将商标等转移过来了。虽然说商业公司支持可以为这些项目提供相对稳定的服务，但许多坚定的 FOSS 开发者不愿意这样，担心项目会偏离“社区驱动、自由开源”的初衷。

毕竟 FOSS 的核心就是社区共建、免费共享，一旦偏向商业化，可能会影响开源的纯粹性。就算他们坚守初心，但谁能保障后来者亦然？尽管如此，Gitea 依然保持着活跃的发展态势，逐渐成为轻量级自托管 Git 服务领域的主流选择，但也有人基于此再次探寻替代方案。

就这样，又有一部分开发者分支了 Gitea，于是 Forgejo 由此诞生。

Codeberg e. V.

Codeberg .e.V 是总部位于德国柏林的非营利组织，它运营基于 Gitea 的代码托管服务（就是 Codeberg）。虽然名气和规模不如 GitHub，但没别的优点，就是开放、自由、民主、共享，因此它在开源社区的名声很好。

Gitea 做的那件事情带来了几个风险：

  • 从社区维护所有转向商业公司控制，存在违背 FOSS 项目原则的风险
  • 商业化可能会导致功能优先级向付费用户倾斜、基础功能受限，甚至出现闭源组件
  • 商业公司的运营决策可能不再对社区完全透明，贡献者的意见和代码贡献不被重视

本来也是 Gitea 重要贡献者的 Codeberg 一早起来天塌了。激怒了日耳曼人是什么后果，略加思索便可知一二。作为长期致力于 FOSS 生态建设的非营利组织，Codeberg 迅速采取行动，在它的支持下 Forgejo 出现了。它的名称源于世界语 “forĝi”（意为“锻造”），象征着打造更纯粹、更强大的社区驱动型开发平台。

它在官网首页就写了：

Brought to you by an inclusive community under the umbrella of Codeberg e.V., a democratic non-profit organization, Forgejo can be trusted to be exclusively Free Software. You can create an account on Codeberg and other instances or download it to self-host your own. It focuses on security, scaling, federation and privacy. Learn more about how it compares with other forges.（由 Codeberg e.V. 这一民主非盈利组织下的包容性社区为您带来，Forgejo 值得信赖，它专注于提供纯粹的自由软件。您可以在 Codeberg 及其他实例上创建账户，或下载它来托管自己的 Forgejo。它专注于安全性、可扩展性、联邦化和隐私保护。了解更多关于它与其他 Forge 的比较情况。）

Take back control of your software development process, self-host your projects and get everyone involved in delivering quality software on the same page.（夺回你对软件开发流程的控制权，自托管你的项目，并让每个人都参与进来，共同确保高质量软件的交付。）

Codeberg 原话是这样的：

Luckily Codeberg is in a unique position to reconnect the Gitea community in one place, independent and out of control of Gitea Ltd. And so we did.（幸运的是，Codeberg 处于一个独特的位置，能够将 Gitea 社区重新聚集在一个独立的地方，不受 Gitea 有限公司的控制。于是我们这么做了。）

本来只是软分叉，对 Gitea 进行修补，但两年后也就是 2024 年，Gitea Ltd 拒绝将资产转移回社区，于是 Forgejo 推出了独立的版本号系统（从 v1.23 直接跳到 v10.0），标志着与 Gitea 的彻底决裂，独立发展。

Codeberg .e.V 表示 Forgejo 的核心目标是“社区掌控、解放软件开发脱离专有工具枷锁”（反正看它的官网也能感受到它对非自由项目的不满），与 Gitea Limited 的商业路线形成鲜明对比。为了起一个好的头，Codeberg 将自己的基础程序从 Gitea 迁移到了 Forgejo。

为了降低迁移难度，Codeberg 提供了 Woodpecker CI、Weblate 等工具，GitHub Pages 也有对应的 Codeberg Pages，还提供了大部分托管平台的预设迁移工具，以及“联邦功能”等组件（我没用过，听说可以让不同实例的用户在实例之间进行协作）（这些估计是微软 GitHub 倒闭了都不会出现的功能）。

虽然它的热度相比于 Gitea 还是较低，但也有了一批忠实的追随者。

Codeberg Pages

等下，它是不是提到 Codeberg Pages 了？

虽然 Codeberg 的节点分布和质量无法与热度高、基础大、背靠微软的 GitHub 相提并论，但也是一个备选方案。不过它的配置方法和 GitHub Pages 又有些区别……

默认域名

Codeberg Pages 现在支持自定义域名的方法是这样的：

  1. 创建一个名为 pages 的公共仓库，以便在主子域名下提供该网站；或是在公共仓库中创建一个名为 pages 的分支
  2. 上传静态资源文件
  3. 使用https://USERNAME.codeberg.page[/REPOSITORY][/@BRANCH]这样的链接访问（自定义域名的方法我们后面再说）

它默认以pages仓库作为 pages 静态资源的存储位置，但你命名为其他的也没问题，只需确保你想展示的资源在其pages分支下（好像这个也不做要求，只是访问的 URL 会多一个路径，略显麻烦）。

点击右上角的加号，创建仓库（如果你愿意创建一个组织统一管理也没有问题），然后按照与 GitHub 相同的方法初始化仓库（记得创建或切换到 pages 分支）：

然后上传文件，等待响应。假如你的用户名或组织名为username、仓库名为reponame、配置了branchname分支，此时你的访问方法是这样的：

域名方案Pages URL对应仓库原始地址
username.codeberg.pagehttps://username.codeberg.pagecodeberg.org/username/pages 仓库
reponame.username.codeberg.pagehttps://username.codeberg.page/reponamecodeberg.org/username/reponame 仓库
branchname.reponame.username.codeberg.pagehttps://username.codeberg.page/reponame/@branchname/codeberg.org/username/reponame/src/branch/branchname

默认是pages仓库或仓库的pages分支，如果不是则需要手动指定仓库名称或分支名称，就像后两个示例一样。

另外，因为使用这种域名对应方法，Codeberg 建议用户不要给用户名或组织名、仓库名、分支名设置带.的名称，而是使用~或_等符号代替。

自定义域名

与 GitHub 不同，这里配置自定义域名不需要在 Codeberg 网页上操作（记得给自己的仓库或组织设置为公开，否则 Codeberg Pages 会找不到这个仓库）。（我建议上一步后先按照它的方法访问一下，就访问Pages URL对应的 URL，确保你的仓库 Pages 是可以使用的）。

仍以刚才的分支为例，我们在username/reponame的branchname分支下需要添加一个.domains文件，里面写上你想要添加的域名（一行一个，不要带协议和斜杠；允许使用#符号注释，也允许空行；第一个为主域名，文件中的所有其他域名都将重定向到第一个域名），然后推送到仓库里。这里 Codeberg 给出了三种方案。我们以page.example.com为例。

哦对了，如果你用的是 Cloudflare 的话，记得设置为仅 DNS（显示灰色云图标）。

CNAME

接下来，进入域名的 DNS 控制台（这里以 Cloudflare 为例），为其添加 DNS 解析。如果你的域名中该子域名没有配置邮箱等服务（即 CNAME 解析是可用的），则这样填写（第三列的数据以全非默认的为例，你可以根据自己的情况酌情修改）：

域名类型数据
page.example.comCNAMEbranchname.reponame.username.codeberg.page

如果要www和根域名都能用，那么这样配置：

域名类型数据
example.comCNAMEbranchname.reponame.username.codeberg.page
www.example.comCNAMEbranchname.reponame.username.codeberg.page

ALIAS

如果不支持 CNAME 解析或用不了 CNAME，那么可以试试 ALIAS 方法。此时配置出来应该是这样的：

域名类型数据
page.example.comALIAScodeberg.page
page.example.comTXTbranchname.reponame.username.codeberg.page

A/AAAA

如果你的域名服务商没有提供 ALIAS 解析记录，也可以使用 A、AAAA 和 TXT 记录，最终大概是这样的（注：Codeberg 的 IP 地址在 2025 年 4 月发生了变更，此处已是最新）：

域名类型数据
page.example.comA217.197.84.141
page.example.comAAAA2a0a:4580:103f:c0de::2
page.example.comTXTbranchname.reponame.username.codeberg.page

如果仍然有问题，可以参考 Using custom domains 和 Troubleshooting 进行排除。另外我觉得这个 issue 也有参考价值：Failed Dependency! “could not obtain repo owner from custom domain” for a custom domain #1013

项目集成

Codeberg 支持使用 Forgejo Actions 工具执行类似于 GitHub Action 的功能。换言之，如果你使用静态网站生成器，如 Jekyll、Hugo 或其他类似工具，你可以将部署集成到 Forgejo Actions 工作流中，而不必在本地执行生成命令。不过目前我暂时没有这个需求（也许以后会加上）。

还有些静态网站生成器，在其资源目录下放置的.domains文件可能不会在执行 deploy 时一并推送。如果这样的话，可以尝试手动将文件复制过去（例如 Hexo 的source和.deploy_git目录下都放一个这样的文件）。

什么是Lorem Ipsum?

Lorem Ipsum，也称乱数假文或者哑元文本， 是印刷及排版领域所常用的虚拟文字。由于曾经一台匿名的打印机刻意打乱了一盒印刷字体从而造出一本字体样品书，Lorem Ipsum从西元15世纪起就被作为此领域的标准文本使用。它不仅延续了五个世纪，还通过了电子排版的挑战，其雏形却依然保存至今。在1960年代，”Leatraset”公司发布了印刷着Lorem Ipsum段落的纸张，从而广泛普及了它的使用。最近，计算机桌面出版软件”Aldus PageMaker”也通过同样的方式使Lorem Ipsum落入大众的视野。
